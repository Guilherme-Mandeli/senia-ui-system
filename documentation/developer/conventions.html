Filosofía General

Senia UI se sustenta en tres principios fundamentales que guían cómo escribimos estilos y estructuramos componentes:

1. CSS Nativo sobre Abstracciones

Priorizamos el uso de estándares web modernos como CSS Custom Properties, Cascading Layers (@layer) y anidamiento nativo. Evitamos el uso excesivo de frameworks de terceros para asegurar un rendimiento de carga óptimo de 100% en navegadores.

2. Componentes Cohesivos con Utilidades de Apoyo

A diferencia del enfoque 100% atómico (Utility-First), defendemos el uso de **Componentes Robustos** que encapsulan los estilos de la marca y se apoyan en **Clases Utility** puntuales únicamente para retocar márgenes o alineaciones. Esto mantiene nuestro HTML limpio y semántico.

3. Flexibilidad Temática Dinámica

Al usar variables CSS globales en lugar de variables compiladas (como en Sass/Less), habilitamos el cambio dinámico de modos claro/oscuro o marcas corporativas en tiempo de ejecución de forma nativa e instantánea.

Nomenclatura BEM Simplificada

Para evitar colisiones de estilos entre diferentes equipos del grupo, adoptamos una variante estructurada de **BEM (Block, Element, Modifier)** combinada con **prefijos de clase funcionales**:

Esquema de Prefijos:

  • c- (Component): Bloques autónomos y reutilizables. Ej: .c-card, .c-button.
  • l- (Layout): Estructuras organizacionales de cuadrícula o contenedores que no aplican estilos estéticos. Ej: .l-grid, .l-container.
  • u- (Utility): Clases de una sola propiedad con alta prioridad. Ej: .u-text-center, .u-mt-vi.

Estructura BEM en HTML

<!-- El Bloque Principal (Componente) -->
<div class="c-card c-card--featured">
  
  <!-- Un Elemento interno adjunto -->
  <h3 class="c-card__title">Título de Tarjeta</h3>
  
  <!-- Otro Elemento interno -->
  <p class="c-card__body">Texto descriptivo...</p>
  
</div>

Colores en Formato HSL

Toda nuestra paleta de colores corporativos se define en el formato **HSL (Hue, Saturation, Lightness)** usando componentes desagregados.

¿Por qué desagregar componentes HSL?

Escribir los colores como variables de componentes individuales nos permite modificar dinámicamente la opacidad (usando la función CSS color-mix o rgba()) o la luminosidad en hovers y estados activos, sin necesidad de declarar una variable nueva para cada caso. 

:root {
  /* Guardamos los componentes puros de H, S, L del Amarillo Senia */
  --color-primary-h: 52;
  --color-primary-s: 100%;
  --color-primary-l: 67%;
  
  /* Creamos el token consumible usando la base */
  --color-primary-500: hsl(var(--color-primary-h), var(--color-primary-s), var(--color-primary-l));
  
  /* Generamos hover oscureciendo la luminosidad */
  --color-primary-600: hsl(46, 86%, 55%); /* Dorado hover */
  
  /* Generamos tintes claros subiendo la luminosidad */
  --color-primary-100: hsl(46, 100%, 92%); /* Fondo suave */
}

1. Familias Tipográficas (font-family)

Establecemos una jerarquía tipográfica unificada que asocia la identidad corporativa de la marca con fuentes de sistema nativas sólidas como salvaguarda secundaria (fallback):

  • Textos de lectura e interfaz: --font-family-body. Resuelve primero a la fuente corporativa Gill Sans, y como caída secundaria a system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, Helvetica, Arial, sans-serif.
  • Títulos y encabezados: --font-family-heading. Resuelve de forma idéntica a Gill Sans con un escalado tipográfico destacado de peso y tracking.
  • Bloques de código y datos monoespaciados: --font-family-mono. Resuelve de manera nativa a JetBrains Mono, Consolas, "Courier New", monospace.

💼 Contexto Serio: Tipografía Lato

Para productos corporativos o secciones con un contexto más formal y serio (ej. informes corporativos, paneles legales, políticas de privacidad), configuramos e integramos la tipografía **Lato**.

El token global es --font-family-lato, y puedes activarlo en cascada de forma nativa e instantánea añadiendo la clase .theme-serious o el atributo data-theme="serious" a cualquier contenedor de tu DOM: 

<!-- Cambiar todo el contenido de una sección a la fuente Lato -->
<div class="theme-serious">
  <h2>Título Formal en Lato</h2>
  <p>Este párrafo se renderizará automáticamente en Lato para dar un tono más formal...</p>
</div>

2. Escala de Tamaños (font-size)

Para garantizar la legibilidad y la accesibilidad en múltiples dispositivos, utilizamos una escala modular tipográfica basada en unidades relativas rem (donde `1rem` equivale por defecto a `16px` del tamaño de raíz del navegador):

Token CSS Valor Relativo Equivalencia Pixels Propósito Común
--font-size-3xs 0.65rem 10.4px Superíndices, pies de imagen muy pequeños.
--font-size-ii 0.75rem 12px Tooltips, subtítulos de tarjetas secundarias, leyendas de inputs.
--font-size-iv 0.875rem 14px Textos de menú lateral, botones primarios, celdas de tablas densas.
--font-size-sm 1rem 16px Cuerpo de texto base (body) en toda la web.
--font-size-md 1.125rem 18px Texto destacado de cabecera, lead paragraphs, subtítulos.
--font-size-vi 1.25rem 20px Títulos de secciones H3, tarjetas destacadas.
--font-size-vii 1.5rem 24px Títulos H2 de página de documentación.
--font-size-viii 2rem 32px Títulos H1 principales en canvas de contenido.
--font-size-ix 3rem 48px Hero Headers en Landing pages corporativas.

3. Mapeo de Pesos Tipográficos (font-weight)

Para evitar deformaciones artificiales de la fuente por parte del navegador (como negritas sintéticas), mapeamos de manera numérica exacta los pesos disponibles en el CDN corporativo:

Gill Sans Light (300)

Reservado para subtítulos ligeros de gran tamaño.

Gill Sans Regular (400)

Párrafos de texto, cuerpo base, menús principales.

Gill Sans Medium (500)

Elementos interactivos con énfasis sutil.

Gill Sans Semibold (600)

El peso por defecto para todos los encabezados (H1-H6).

Gill Sans Bold (700)

Negrita estándar para textos con fuerte importancia semántica.

Gill Sans Heavy (900)

Máxima contundencia tipográfica para textos Hero.

4. Carga Inteligente de Fuentes (Performance)

Estándar LCP

Carga diferida bajo demanda (Lazy Loading Nativo)

A nivel de hoja de estilos corporativa, registramos las variantes de la tipografía Gill Sans usando reglas @font-face. Sin embargo, los navegadores modernos **solo descargan el archivo de la fuente del CDN si existe al menos un elemento en el DOM usando ese peso específico**.

Esto significa que al usar por defecto únicamente los pesos `400` y `600` en tu layout, las variantes `Light (300)`, `Bold (700)` o `Heavy (900)` **nunca consumirán ancho de banda de tu usuario**, optimizando al máximo el LCP (Largest Contentful Paint) y la velocidad de carga inicial.

5. Guía de Buenas Prácticas Tipográficas

  • Heredar en lugar de forzar: Evita declarar familias tipográficas ad-hoc en tus clases. Usa siempre var(--font-family-body) o simplemente confía en la herencia nativa de etiquetas.
  • Unidades Relativas: Nunca uses `px` para propiedades de font-size o line-height. Define los tamaños con variables del sistema (ej. var(--font-size-iv)) para soportar el zoom de accesibilidad nativo del navegador.
  • Evitar deformaciones: Nunca apliques pesos no soportados como font-weight: 800 o font-weight: bold sobre Gill Sans si no has cargado el peso correspondiente en CSS, ya que forzará al navegador a sintetizar una negrita borrosa artificial.

Clases de Utilidad de Apilamiento (Z-Index Utilities)

En lugar de escribir estilos embebidos, puedes aplicar directamente las clases de utilidad estructuradas bajo el patrón .u-z-[capa] en tu HTML.

Clase de Utilidad Valor Resuelto
.u-z-hide -1
.u-z-auto auto
.u-z-base 0
.u-z-dropdown 1000
.u-z-sticky 1020
Clase de Utilidad Valor Resuelto
.u-z-fixed 1030
.u-z-modal-backdrop 1040
.u-z-modal 1050
.u-z-popover 1060
.u-z-tooltip 1070
Ejemplo práctico en HTML: 
<!-- Cabecera flotante e interactiva combinando utilidades -->
<header class="u-fixed u-top-0 u-w-full u-z-sticky u-p-iv">
  <span>Cabecera fija sobre contenido</span>
</header>
